# Adamrit: End-to-End Hospital Management System for an ESIC-Empanelled Hospital

## The Problem

An ESIC-empanelled hospital was running patient registration, OPD/IPD encounters, lab orders, pharmacy dispensing, billing, and ledger accounting across disconnected tools and paper-based processes. Scheme-specific billing (ESIC, CGHS, PMJAY, MJPJAY, corporate) required manual cross-referencing, and there was no single system of record staff could trust. Discharge documentation, pharmacy clearance, and gate pass generation were handled separately, creating coordination gaps at every handoff.

## What We Built

- **Patient lifecycle management** — Registration (including self-check-in kiosk and bulk import), scheme membership linkage (ESIC / CGHS / PMJAY / MJPJAY / corporate / private), and identity verification, all tied to a master patient record that persists across visits.
- **OPD → IPD → Discharge workflow** — Single visit record as the anchor point; clinical staff capture diagnoses (ICD-10 coded), complications, treatment sheets, and shifting/admission notes from one surface. Discharge clearance fields (nurse, pharmacy, gate pass) are tracked on the same row.
- **Integrated orders** — Lab worklists, radiology orders (CT/MRI, CathLab), and pharmacy prescriptions all attach to the visit. Lab supports home collection. Pharmacy enforces First Expiry First Out and blocks dispensing of expired batches.
- **Frozen final bill with audit integrity** — `FinalBill.tsx` is SHA256-locked at build time; a failed hash check blocks Vercel deployment. Scheme-specific billing paths (Yojna bill, corporate submission, no-deduction letter) branch from the same billing chain. All financial data queries live bills and line items directly — no derived summary table that can go stale.
- **Double-entry accounting + Tally export** — Vouchers, ledger groups, and payment allocations are maintained in-app. A Vercel serverless function handles Tally export, keeping accounting reconciliation inside the same system rather than a separate spreadsheet.
- **Agent-grounded corpus** — SOPs, schema documentation, and clinical workflow notes are stored in a vault, synced to Supabase Storage, chunked and embedded via an Edge Function, and made available to operational agents scoped by department and sensitivity. Staff added mid-project (Sahil, Anirudh, and four BCA interns) onboarded against this corpus rather than tribal knowledge.

## The Result

- One system of record replaced disconnected tools across registration, clinical, pharmacy, billing, and accounts — staff query a single source rather than reconciling across departments.
- The SHA256 build guard has prevented unauthorised changes to the final bill page, reducing billing integrity risk on every deployment.
- Pharmacy batch enforcement (FEFO + expiry block) reduced the manual effort required to verify stock before dispensing.
- Tally export via serverless function eliminated manual ledger re-entry at end of day.
- Onboarding new team members reduced to documented SOP review rather than shadowing senior staff.

## Murali's Note

The non-obvious decision was locking `FinalBill.tsx` at the build pipeline level — hospitals lose money not because software is wrong on day one, but because well-meaning developers quietly adjust bill calculations over time; the hash check makes that impossible to do accidentally.

## Stack

Supabase (Postgres + RLS + Storage + Edge Functions), React, TypeScript, Vite, Vercel (static hosting + serverless functions), Twilio, Tally export integration, OpenAI-compatible embeddings (DeepSeek), ICD-10 coding
